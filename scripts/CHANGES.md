# Meteor Client Chinese Localization Patches

## Overview

This document describes all modifications made in this branch to the original Meteor Client source code for Chinese localization support. All code and documentation in this branch were generated by AI.

---

## 1. Translation Runtime Infrastructure

### Modified files:

- `src/main/java/meteordevelopment/meteorclient/utils/Utils.java`
- `src/main/java/meteordevelopment/meteorclient/MeteorClient.java`
- `src/main/java/meteordevelopment/meteorclient/commands/Command.java`
- `src/main/java/meteordevelopment/meteorclient/gui/themes/meteor/MeteorGuiTheme.java`

### Changes

**Utils.java** — Three static members added:

| Member | Purpose |
|--------|---------|
| `TRANSLATIONS` (`HashMap<String,String>`) | In-memory translation map |
| `loadTranslations()` | Reads `zh_cn_full.json` from classpath, populates the map |
| `tr(String)` | Returns translated string, falls back to English |
| `trIfPresent(String)` | Returns translated string or null |

**MeteorClient.java** — `onInitializeClient()` now calls `Utils.loadTranslations()` BEFORE `ReflectInit.registerPackages()`, so that modules/settings are constructed with translated strings from the start.

**Command.java** — Constructor wraps `title` and `description` with `Utils.tr()`.

**MeteorGuiTheme.java** — The following widget factory methods wrap their String parameter with `Utils.trIfPresent()`: `window()`, `label()`, `horizontalSeparator()`, `button()`, `confirmedButton()`, `tooltip()`, `section()`. This ensures all GUI text elements display Chinese without needing per-call changes.

## 2. Translation Data

### Added files:

- `scripts/translations.json` — Master translation table (3210 entries)
- `src/main/resources/assets/meteor-client/lang/zh_cn_full.json` — Runtime language file (generated)

### Modified files:

- `scripts/translate.py`
- `.github/workflows/translate.yml`

### Workflow

1. `translate.py` scans all 949 Java source files, extracts 3213 unique English strings via regex patterns.
2. Loads `translations.json` (hardcoded translations, built offline by AI).
3. For any string not found in the table, falls back to Google Translate API.
4. Writes output to `zh_cn_full.json`.

### Translation Statistics

| Source | Count |
|--------|-------|
| Hardcoded direct translations (translations.json) | 3213 |
| Google API fallback (untranslated at CI time) | ~0 (all hardcoded) |
| Still English (intentional: +, -, :, {} {} etc.) | 7 |

No pattern rules are used. Every translation is a direct English→Chinese string pair.

## 3. Font & CJK Rendering

### Modified files:

- `src/main/java/meteordevelopment/meteorclient/renderer/text/Font.java`
- `src/main/java/meteordevelopment/meteorclient/renderer/text/CustomTextRenderer.java`
- `src/main/java/meteordevelopment/meteorclient/systems/config/Config.java`
- `src/main/java/meteordevelopment/meteorclient/systems/hud/Hud.java`

### Changes

**Config.java** — `customFont` default stays `true`. No renderer change.

**Hud.java** — Same, `customFont` default stays `true`.

**Fonts.java** — Added "WenQuanYi Micro Hei" (subset, 309 KB) to `BUILTIN_FONTS` and set it as the new default. This font contains all CJK glyphs needed by the translations (946 unique CJK characters + Latin/Greek/Cyrillic).

**Font.java** — Added `CJK Unified Ideographs` glyph range (U+4E00–U+9FFF, 20992 codepoints) to the STB TrueType glyph packing. The WenQuanYi font has these glyphs, so they are now packed into the custom renderer's texture atlas and render correctly.

No other font changes. The same `CustomTextRenderer` handles both Latin and CJK text with consistent metrics.
